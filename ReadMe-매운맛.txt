개발 규칙
1. 브랜치 작명은 
	1) 기능 개발 시 : feacher/기능요약명
	2) 출시 버전 브랜치 : release/기능요약명
	3) 핫픽스 브랜치 : hotfix/기능요약명
	이지만, 어차피 기능 브랜치만 만들거니 작명은 feacher/기능요약명 으로 해주면 된다.

2. 커밋 및 푸쉬
자신이 하루 동안 작업한 내용을 누군가와 함께 사용하는 브랜치에 커밋하거나 푸쉬하려 할 때는, 당사자에게 알린다.
해당 인원이 무언가 개발중이라면, 겹치는 수정 파일이 없는지 확인하고, 최대한 겹치지 않는 것이 좋지만 만약 불가피하게
겹치게 되었다면, 겹친 내용을 합친 새 파일을 만들어 한 사람이 커밋 및 푸쉬하거나, 따로 업로드한다.

만약 오류가 뜨든 말든 멍청하게 이미 푸쉬를 해버렸다! 한다면, 
먼저 팀장에게 알린다. 그러면 해결해줄거다.
만약 더 일이 커질지도 모르지만 내 힘으로 해결하고 싶다! 면,
먼저 자신이 올린 수정파일과, 자신이 올린 수정파일의 이전 버전을 대조하고 비교한 뒤 새로 만들어 커밋해야 할 것이다.

만약 안되면 뭐.. 다시 만들어야지..

3. 머지
머지는 보통 팀장이 합니다. 
당신네들은 권한이 없을겁니다.
브랜치 병합 하는 행위를 머지한다고 하는데,
머지가 필요할 때는 팀장을 부르세요.

4. main 브랜치
절대 건드리지 마시고, 절대 무언가를 커밋 하지도 마시고, 무언가 머지 하려는 행위도 하지 마세요.
살해 당할 수 있습니다.
